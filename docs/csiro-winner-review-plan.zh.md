# CSIRO 冠军 `0.67` 题解深度 Review 方案

## 使用口径

这份文档不是正式 review 结论稿，而是一份“深度技术复盘方案文档”。目标是把冠军题解、当前仓库能力和当前实验现状整理成一个可执行的分析框架，供后续直接写正式 review 和安排实验优先级。

本文固定使用以下三类标签，避免把判断写混：

- 证据标签：
  - `[事实]`：题解原文、仓库代码、当前文档里能直接核到的内容
  - `[推断]`：基于当前结果或实现状态做出的解释
  - `[假设]`：需要后续实验验证，当前不能当事实写
- 视角标签：
  - `[OOF]`
  - `[LB/PB]`

本文固定基于以下输入：

- `[事实][LB/PB]` 本地保存的冠军题解 `csiro-biomass.html`
- `[事实][代码]` 当前仓库实现
- `[事实][OOF]` 当前仓库内仍能直接核验的实验文档与指标
- `[假设][OOF]` 当前实验主线最新分数输入：
  - `dinov3-vitl-896-timm = 0.5031`
  - `dinov2-vitg-518 = 0.4546`
  - `dinov2-vitl-518 = 0.4347`
  - `siglip-so400m-384 = 0.0695`
  - `siglip-so400m-448 = 0.0356`
  - `dinov3-vitl-1024-timm = 0.4914`
  - `dinov3-vithplus-896-timm = 0.4873`
  - `dinov2-vitl-reg4-518 = 0.4687`
  - `dinov2-vitg-reg4-518 = 0.4605`

说明：

- `[事实]` 当前仓库内没有与上述最新 9 条基线一一对应的 `artifacts/server/*/oof_summary.json` 和 `oof_metrics.csv`。
- `[假设][OOF]` 因此本文把这 9 条分数视为“当前实验结论输入”，用于组织 review 结构和优先级，不把它们伪装成已本地核验的原始产物。
- `[推断]` 正式 review 落稿前，仍应补齐这些实验的证据路径和 per-target 指标表。

## 1. 题解要点抽取

### 1.1 数据处理

- `[事实][LB/PB]` 冠军题解明确写出：CV 按 `state + sampling date` 联合划分三折，然后训练 single-fold model。
- `[事实][LB/PB]` 题解明确写出：原始宽图被切成 left/right 两个视图，送入同一个 backbone。
- `[事实][LB/PB]` 题解明确写出：左右子图在切分后独立做增强。
- `[事实][LB/PB]` 题解明确写出：随机缩小后用黑边补齐，以模拟不同相机焦距和视角。
- `[推断]` 这个黑边缩放不是普通“增强堆料”，而是显式在补数据采集差异。
- `[推断]` 冠军题解没有展开说清楚 CV 是否还有更细的分层约束，但 `state + sampling date` 至少说明他们在主动压制泄漏风险。

### 1.2 Backbone

- `[事实][LB/PB]` 题解明确写出：主 backbone 用 `DINOv3`。
- `[事实][LB/PB]` 题解明确写出：左右特征 concat 后用单层 multi-head self-attention 做交互，再经 MLP 输出融合特征。
- `[事实][LB/PB]` 题解明确写出：后期在算力有限时，更倾向“增大输入尺寸”而不是“增大模型体量”。
- `[事实][LB/PB]` 题解明确写出：最佳单模主线通常是 `1024 pix + DINO-Large`。
- `[推断]` 冠军方案真正押注的是“大输入 + 强 backbone + 双流一致性”，不是“无脑堆更大模型”。

### 1.3 Head 设计

- `[事实][LB/PB]` 题解明确写出：使用 `5` 个独立回归头，每个头都是三层线性层。
- `[事实][LB/PB]` 题解明确写出：另外加了 `5` 个区间分类头。
- `[事实][LB/PB]` 题解明确写出：每个 target 被分成 `7` 个区间。
- `[事实][LB/PB]` 题解明确写出：分类头相对纯回归带来了约 `+0.03` 的 `LB/PB` 增益，且和他们的 `CV` 判断一致。
- `[推断]` 分类头对冠军不是边角料，而是单模阶段的主增益项之一。
- `[推断]` 这一步很可能尤其帮助了长尾目标和高低值混杂目标的排序稳定性。

### 1.4 损失

- `[事实][LB/PB]` 题解明确写出：最终提交模型用 `SmoothL1` 做回归，用 `CrossEntropy` 做区间分类。
- `[事实][LB/PB]` 题解明确写出：loss 按比赛目标权重加权。
- `[事实][LB/PB]` 题解明确写出：他们还试过 `epsilon-insensitive L1`，在 `PB` 上拿到过最好单模。
- `[推断]` 最终提交没有上 `epsilon-insensitive loss`，说明它更像单模 PB 定向优化，而不是被稳定证明的主线方案。

### 1.5 训练 schedule

- `[事实][LB/PB]` 题解明确写出：采用两阶段训练。
- `[事实][LB/PB]` Stage 1 冻结 DINO，只训练 fusion 层和 MLP heads。
- `[事实][LB/PB]` Stage 2 解冻全模型 fine-tune。
- `[事实][LB/PB]` 题解明确写出：`DINO-Large` 和 `DINO-Huge` 在他们设置下表现接近。
- `[推断]` 这说明冠军团队在单模阶段已经观察到“模型大小收益接近饱和”。

### 1.6 Pseudo-label / Online Training

- `[事实][LB/PB]` 题解明确写出：测试时 online training 带来 `> 0.02` 增益。
- `[事实][LB/PB]` 题解明确写出：先挑 `4` 个在 `LB` 上表现好且有差异性的单模来产 pseudo-label。
- `[事实][LB/PB]` 题解明确写出：第一轮 pseudo 后训练 `2` 个 `DINO-Large`。
- `[事实][LB/PB]` 题解明确写出：第二轮 pseudo 后训练 `2` 个 `DINO-Base`。
- `[事实][LB/PB]` 题解明确写出：pseudo 训练时不再做 `CV`，而是使用全训练集 + 测试集，并取最后多个 epoch 的 `SWA ensemble`。
- `[事实][LB/PB]` 题解明确写出：pseudo 后还会回到训练集继续训几轮，防止模型过拟合 pseudo-label。
- `[推断]` 这里真正的关键不是“有 pseudo 就行”，而是“teacher 选择 + 两轮迭代 + SWA + anti-overfit 回训”这一整套链路。

### 1.7 Ensemble

- `[事实][LB/PB]` 题解明确写出：最终融合 `2` 个 `DINO-Large` 和 `2` 个 `DINO-Base`。
- `[事实][LB/PB]` 题解明确写出：融合权重为 `0.4 / 0.6`。
- `[事实][LB/PB]` 题解原文只写了“performed well at LB with variability”，没有展开 teacher 去相关的具体算法。
- `[推断]` “with variability” 本质上就是 teacher 多样性筛选，只是题解没把实现细节写透。
- `[假设]` 冠军队很可能做了经验式去相关，而不是完全靠机械 top-k 取分数。

### 1.8 后处理

- `[事实][LB/PB]` 题解明确写出：`Dry_Clover_g *= 0.8`。
- `[事实][LB/PB]` 题解明确写出：`Dry_Dead_g > 20` 时乘 `1.1`，`Dry_Dead_g < 10` 时乘 `0.9`。
- `[事实][LB/PB]` 题解明确写出：`GDM_g = 0.5 * out_gdm + 0.5 * (out_green + out_clover)`。
- `[事实][LB/PB]` 题解明确写出：`Dry_Total_g = 0.5 * out_total + 0.5 * (green + clover + dead)`。
- `[事实][LB/PB]` 在评论区，冠军作者补充说这些后处理“部分参考 open-source notebooks，部分来自 CV 和 LB score”。
- `[推断]` 这说明 postprocess 是规则型经验修正，不是从严格物理约束自动推出来的。
- `[推断]` `Dry_Dead_g / Dry_Clover_g` 被单独修正，本身就是一个强信号：这两个目标很可能是他们也明确感知到的短板来源。

### 3.3 这张矩阵应输出的核心结论

- `[推断]` 当前仓库和冠军在“模型骨架”上并不远，真正差距不在双流、attention 融合、多头设计这些基础结构。
- `[推断]` 当前更大的缺口在“实验闭环完整度”，尤其是：
  - teacher 选择与去相关
  - 多模型 pool
  - pseudo-label / online training
  - 短板 target 的显式修正链路
- `[推断]` 因此后续实验不应再把大部分预算继续投入“只换更大 backbone / 更高分辨率”。

## 4. 差距来源归因

### 4.1 单模 vs 多模型

- `[事实][LB/PB]` 冠军最终不是单模，而是 `2` 个 `DINO-Large` + `2` 个 `DINO-Base` 的融合。
- `[推断][OOF]` 我们当前 `0.50` 左右主要还是单模视角。
- `[推断][LB/PB]` 冠军 `0.67` 不能被理解为“某个单模天然高出我们 0.17”，而更像是“强单模 + 多模型 pool + 测试时链路”共同叠加后的结果。
- `[推断]` 这是解释分差的第一层主因。

### 4.2 Pseudo-label / Online Training 缺失

- `[事实][LB/PB]` 冠军题解明确写出 online training 带来 `>0.02` 增益。
- `[推断][OOF]` 当前仓库虽有 pseudo 代码，但没有形成已验证强主线。
- `[推断][LB/PB]` 如果这条链路缺失，那么我们和冠军至少会少掉一块非边际增益。
- `[推断]` 这是解释 `0.67` 与 `0.50` 之间差距的高优先级原因。

### 4.3 目标短板未修复

- `[事实][OOF]` 当前本地可核的 `dinov2-vitl-518` 已经显示 `Dry_Dead_g / Dry_Clover_g` 明显拖分。
- `[事实][LB/PB]` 冠军 postprocess 直接对 `Dry_Dead_g` 和 `Dry_Clover_g` 做了规则修正。
- `[推断][OOF]` 即使当前主线换成 `dinov3`，这两个 target 更可能仍是拖分核心。
- `[推断]` 当前分差不是“五个目标都差不多落后”，而更像是“少数短板目标持续拖住总体上限”。

### 4.4 输入尺寸 / 模型大小收益已饱和

- `[事实][LB/PB]` 冠军明确写出：他们后期更偏向增大输入尺寸，而不是增大模型体量。
- `[假设][OOF]` 当前结果输入显示：
  - `1024` 没超过 `896`
  - `H+` 没超过 `ViT-L`
- `[推断][OOF]` 这说明我们当前阶段继续只做“大模型 / 大分辨率”扫参，边际收益已经很有限。
- `[推断]` 这部分可以解释“为什么我们没靠更大 backbone 自动逼近冠军”。

### 4.5 后处理与融合策略不足

- `[事实][LB/PB]` 冠军最终提交不仅有多模型融合，还有规则后处理。
- `[事实][代码]` 仓库已实现近似同款 postprocess，也支持加权 ensemble、TTA、checkpoint averaging。
- `[推断][OOF]` 问题不在“代码没有”，而在“还没把这些能力编排成被证明有效的主线策略”。
- `[推断]` 这是第二层差距，不如 pseudo / 多模型 pool 核心，但绝不是可忽略项。

### 4.6 Leaderboard / Test 分布与 OOF 的差异

- `[事实][LB/PB]` 冠军很多选择是由 `CV + LB` 联合驱动的，尤其是后处理和 teacher 选择。
- `[推断][OOF]` 我们当前拿 `OOF ≈ 0.50` 去直接对比冠军 `LB/PB ≈ 0.67`，口径并不对齐。
- `[推断][LB/PB]` 一部分差距可能来自 test 分布、榜单反馈、以及 test-time chain 对 leaderboard 的适配。
- `[推断]` 不能把 `0.17` 全部归因给训练主干本身。

### 4.7 本节固定总判断

- `[推断][OOF + LB/PB]` “为什么冠军能到 `0.67`，而我们当前停在 `0.50` 左右”的更合理回答是：
  - 不是我们主干完全错了
  - 而是我们还停在“强单模 + 局部能力存在”的阶段
  - 冠军已经走到“强单模 + 多模型 pool + teacher 多样性 + online training + 规则修正”的完整闭环

## 5. 后续实验优先级

### 5.1 优先做

| 优先级 | 方向 | 原因 |
| --- | --- | --- |
| P0 | 做基于最新主线的 teacher pool 与去相关筛选 | 当前仓库已有能力，且最能低成本接近冠军的 “with variability” 逻辑 |
| P0 | 把 `dinov3-vitl-896-timm` 扩成多 seed / 多模型 pool | 当前最强单模已明确，下一步应从单模上限转向模型池 |
| P0 | 跑通并量化 pseudo-label / online training 主链路 | 冠军明确给出 `>0.02` 量级增益，这是主差距来源之一 |
| P0 | 围绕 `Dry_Dead_g / Dry_Clover_g` 做定向修复实验 | 这是当前最像“结构性短板”的 target 组 |
| P0 | 把 postprocess 放到最新主线和 ensemble 上做真实 ablation | 仓库已有同款规则，但缺主线证据 |

### 5.2 可选做

| 优先级 | 方向 | 原因 |
| --- | --- | --- |
| P1 | 在已筛过的 teacher pool 上试 `TTA + checkpoint averaging` 组合 | 仓库已有能力，但更像融合增益，不是主差距第一来源 |
| P1 | 用 `dinov2` / `siglip` 仅作为异构补充而非主线 | 可能提供 decorrelation 价值，但不应重新投入主线预算 |
| P1 | 补齐最新各实验的 `oof_metrics.csv` 与 per-target 表 | 这是正式 review 落稿前的证据工程，不是增益来源本身 |

### 5.3 暂不做

| 优先级 | 方向 | 原因 |
| --- | --- | --- |
| P2 | 继续主攻更大 backbone | 当前结果已显示不自动涨分 |
| P2 | 继续主攻更高分辨率 | `1024` 当前未超 `896`，收益不稳定 |
| P2 | 把 `siglip` 重新拉回主线 | 当前失败线过弱 |
| P2 | 先做复杂外部半监督方案 | 冠军自己也写了多种 semi-supervised 没优于直接 hard pseudo |
| P2 | 先做多分辨率 patch、ranking loss、物理约束重构 | 冠军题解已给出“不稳定或最终榜单效果不好”的负面证据 |

### 5.4 文末应固定输出的一句话结论

- `[推断]` 下一阶段最值得做的不是继续卷单模 backbone，而是把当前已经具备的代码能力补成冠军那种“teacher 筛选 + 多模型 pool + online training + 短板 target 修正”的完整实验闭环。
